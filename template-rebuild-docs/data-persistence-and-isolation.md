# Data Persistence And Isolation

## Mục đích

Tài liệu này định nghĩa persistence model, schema groups, migration rules, tenant isolation, knowledge/RAG storage, memory governance, và data lifecycle.

## Đối tượng đọc

Backend engineer, database engineer, AI engineer, security reviewer, QA, và operator.

## Storage Strategy

Use PostgreSQL as the transactional source of truth.

Template already uses PostgreSQL, SQLModel, Alembic, LangGraph checkpoint tables, and pgvector-backed mem0 memory. Agent Support should extend this with tenant-aware domain tables and keep vector retrieval behind a provider contract.

Recommended stores:

| Store | Purpose |
| --- | --- |
| PostgreSQL | Tenants, auth/membership, config, platform mapping, chat events, agent runs, audit, source metadata, sync jobs, policies. |
| pgvector | Template-native vector backend for early memory/RAG if it passes isolation and citation tests. |
| Qdrant optional | Dedicated vector backend later if scale, filtering, operations, or isolation needs justify it. |
| Valkey/Redis | Cache, rate limiting, optional streams/outbox transport. Not audit source. |
| Object storage optional | Raw source snapshots/uploads, large artifacts. |
| Secret manager | Provider/API credentials and tenant tool secrets. |

## SQLModel And Alembic

Template uses SQLModel. Rebuild should not blindly copy an SQLAlchemy-only rule from a previous implementation.

Use SQLModel pragmatically if staying with the template:

- Keep API schemas separate from persistence models when external contracts differ.
- Do not return ORM objects directly.
- Use Alembic for every schema change.
- Use raw SQL in Alembic for RLS, roles, grants, indexes, extension setup, and DB-specific policies.
- Review autogenerated migrations manually.

If the team wants pure SQLAlchemy, decide before domain schema work begins. Mixing patterns without a decision will slow the rebuild.

## Tenant Isolation Model

Preferred:

- PostgreSQL RLS or equivalent DB-enforced tenant policies for tenant-owned tables.
- Application context sets tenant id for tenant-scoped operations.
- Platform/operator sessions remain separately authorized and audited.

Required regardless of implementation:

- Cross-tenant read/write denial tests.
- Least-privileged role tests.
- Tenant filters in every repository/service query.
- Code review gate for new tenant-owned table.
- No trusted tenant id from untrusted request body.

## Core Tables

### Tenant And Auth

```text
tenants
tenant_memberships
tenant_roles
service_principals
tenant_config_versions
```

Key fields:

- tenant id,
- slug/display name,
- status,
- persona/model/moderation/tool defaults,
- config version,
- timestamps.

### Platform And Messaging

```text
tenant_platforms
adapter_credentials
chat_events
delivery_outbox
delivery_receipts
```

Key rules:

- adapter resolves tenant through platform mapping;
- inbound idempotency prevents duplicate processing;
- outbound links to inbound event and agent run;
- delivery records support retry and audit.

### Agent Runtime

```text
agent_runs
agent_run_steps
model_calls
graph_checkpoint_metadata
```

Key rules:

- run links to tenant, trace id, input event, graph version;
- steps record node status/latency/redacted summary;
- model calls record provider/model/prompt/config/policy versions;
- checkpointer state is useful for resume/replay but not the only audit evidence.

### Knowledge And RAG

```text
knowledge_sources
knowledge_source_versions
knowledge_documents
knowledge_chunks
knowledge_sync_jobs
knowledge_candidates
knowledge_ingest_audit
```

Key rules:

- PostgreSQL owns source metadata and activation.
- Vector backend stores retrieval payload, not product policy.
- New source version is not active until verification passes.
- Candidate knowledge cannot self-approve.

### Capabilities And Tools

```text
plugin_manifests
plugin_capabilities
tenant_capabilities
tenant_tool_policies
tenant_credential_handles
tool_calls
sub_agent_invocations
```

Key rules:

- capability version recorded on run/tool records;
- hot-disable rechecked before each execution;
- credential material lives outside config table.

### Moderation And Review

```text
moderation_decisions
moderation_actions
review_queue_items
policy_versions
```

Key rules:

- shadow/propose/enforce mode is explicit;
- destructive actions have idempotency key;
- policy version stored with decision/action.

### Audit

```text
audit_events
```

Audit should cover admin mutations, source lifecycle, capability changes, tool attempts, moderation, operator incident access, and security-sensitive operations.

## Vector Payload Contract

Any vector backend must support equivalent metadata:

```json
{
  "tenant_id": "uuid",
  "source_id": "uuid",
  "source_version_id": "uuid",
  "document_id": "uuid",
  "chunk_id": "uuid",
  "visibility": "public|private|internal",
  "source_uri": "string",
  "source_title": "string",
  "section_path": ["string"],
  "locale": "en",
  "content_hash": "sha256",
  "active": true,
  "updated_at": "timestamp"
}
```

Retrieval must filter by:

- tenant id,
- active source version,
- visibility allowed for current channel/user/policy,
- optional source allowlist,
- optional locale,
- deleted/tombstoned state.

## Knowledge Sync Lifecycle

```text
source_registered
-> sync_requested
-> fetch_started
-> normalized
-> chunked
-> embedded
-> vector_upserted
-> verified
-> source_version_active
```

Failure states:

- fetch failed,
- parse failed,
- secret/PII blocked,
- embed failed,
- vector write failed,
- partial verification failed,
- stale/tombstoned.

Partial sync must not become visible.

## Source Activation Strategy

1. Create immutable source version.
2. Parse/chunk/embed under that source version.
3. Upsert vector records with source version payload.
4. Verify counts and sample retrieval through provider.
5. Mark source version active in PostgreSQL.
6. Query path uses active version list from PostgreSQL or equivalent.
7. Deactivate/tombstone old chunks only after new active version is available.

## Memory Governance

Template mem0 auto-extracts memories from conversations. For Agent Support this can be risky unless governed.

Recommended stance:

- Disable or restrict automatic long-term memory for tenant community messages until retention, prompt-access, and review policy are accepted.
- Do not use generic mem0 memory as official knowledge.
- Use source-backed knowledge pipeline for RAG answers.
- Keep conversation memory as recent context or summaries with tenant retention policy.

Memory categories:

| Memory | Storage | Use |
| --- | --- | --- |
| Runtime state | LangGraph checkpoint | Resume/replay current graph. |
| Recent conversation | Chat events/summaries | Context continuity. |
| Config/policy | PostgreSQL | Hydration. |
| Knowledge | pgvector/Qdrant + PostgreSQL metadata | Citation-backed support. |
| Profile | PostgreSQL later | Restricted/policy-gated personalization. |
| Audit/replay | PostgreSQL | Incident/compliance. |
| Feedback/eval | Redacted datasets | Regression and quality. |
| Secrets | Secret manager/handles | Never prompt-visible. |

## Migration Rules

Before merging schema changes:

- Migration upgrade from empty database.
- Downgrade or documented forward-only rollback.
- RLS/equivalent policies included for tenant-owned tables.
- Grants/roles reviewed.
- Indexes for tenant id, trace id, source version, platform idempotency, and audit queries.
- No business logic inside migrations.
- Tests prove existing data paths still work if applicable.

## Validation Gates

Minimum:

```bash
make install
make migrate
pytest
ruff check .
pyright app evals
detect-secrets scan --baseline .secrets.baseline
```

Adapt exact commands to the final template Makefile/CI. For product-specific gates add:

- cross-tenant SQL denial,
- vector tenant/visibility denial,
- source update/delete/tombstone,
- disabled tool denial,
- audit record creation,
- migration downgrade,
- redaction tests.
