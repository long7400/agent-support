# Persistence Strategy

## Mục đích

Định nghĩa persistence model, storage choices, isolation model, memory governance, retention/lifecycle sau khi chốt decisions.

## Đối tượng đọc

Backend engineer, database engineer, AI engineer, security reviewer, QA, operator.

## Storage Strategy

PostgreSQL là transactional source of truth.

| Store | Purpose |
| --- | --- |
| PostgreSQL | Tenants, auth/membership, config, platform mapping, chat_events, outbox, agent_runs, audit, source metadata, sync jobs, policies. RLS enforced. |
| Qdrant | Vector backend (ADR-001), external, app-layer tenant filter. |
| Redis/Valkey | Cache + rate limit only. **KHÔNG** outbox, KHÔNG audit source. |
| Object storage | Raw source snapshots/uploads (Markdown), large artifacts. |
| GCP Cloud KMS | Master key cho envelope encryption (ADR-006/008). |
| Langfuse (self-host) | Trace backend (ADR-007), redacted, không phải audit source. |

> Lưu ý: vì chốt Qdrant (ADR-001), pgvector extension **không bắt buộc** cho RAG. mem0/pgvector chỉ giữ nếu cần generic memory — không đồng nhất với curated knowledge.

## ORM: SQLAlchemy 2.0 (ADR-004)

Migrate template SQLModel → **SQLAlchemy 2.0 thuần + Pydantic v2 riêng** ở Phase 0.

Rules:
- Tách persistence model (ORM) khỏi API DTO (Pydantic). Không return ORM object trực tiếp.
- Async SQLAlchemy 2.0 + asyncpg.
- Alembic cho mọi schema change.
- Raw SQL trong Alembic cho RLS, roles, grants, indexes, extension setup.
- Review autogenerated migration thủ công.

Lý do migrate: RLS pattern production-grade không có cho SQLModel; ~15-20 domain tables; mix 2 ORM = nợ kỹ thuật. Chỉ template auth.py + session model chạm khi migrate.

## Tenant Isolation: PostgreSQL RLS (ADR-002)

RLS toàn diện cho mọi tenant-owned table. Chi tiết SQL pattern: [migration-rules.md](migration-rules.md). Threat coverage: [../03-security/threat-model.md](../03-security/threat-model.md).

Core mechanism:
- App role không phải owner (least-privileged, không BYPASSRLS).
- `SET LOCAL app.current_tenant = '<uuid>'` trong `async with db.begin()` (BẮT BUỘC trong transaction — ngoài tx = leak qua pool).
- Operator role riêng với `BYPASSRLS` cho incident response, audit mọi access.
- pgbouncer (nếu dùng) phải session-pool mode, không transaction-pool.

LangGraph checkpointer: include `tenant_id` trong checkpoint metadata + filter app-side (checkpointer không tự set tenant context).

Qdrant: không có RLS → mandatory app-layer tenant filter trong retrieval contract. Isolation test là release gate.

## Memory Governance

Không dùng một generic memory bucket. Template mem0 auto-extract memories từ conversation → risk cho community.

Stance:
- Disable/restrict automatic long-term memory cho tenant community messages cho đến khi retention + prompt-access + review policy được chấp nhận.
- mem0 generic memory KHÔNG phải official knowledge.
- Source-backed knowledge pipeline (Qdrant) cho RAG answers.
- Conversation memory = recent context/summaries với tenant retention policy.

| Memory | Storage | Use |
| --- | --- | --- |
| Runtime state | LangGraph checkpoint | Resume/replay current graph. |
| Recent conversation | chat_events/summaries | Context continuity. |
| Config/policy | PostgreSQL | Hydration. |
| Knowledge | Qdrant + PostgreSQL metadata | Citation-backed support. |
| Profile | PostgreSQL (later) | Restricted/policy-gated. |
| Audit/replay | PostgreSQL | Incident/compliance. |
| Feedback/eval | Redacted datasets | Regression/quality. |
| Secrets | KMS + credential table | Never prompt-visible. |

## Retention Policy (ADR Decision 13)

Default retention (per-tenant override qua `tenants.retention_policy_json`, floor enforced):

| Data | Default | Reasoning |
| --- | --- | --- |
| `chat_events` | 90 days rolling | Đủ incident replay, không phình DB. |
| `agent_runs` + `run_steps` | 180 days | Replay window dài để debug pattern. |
| `model_calls` | 30 days detail + lifetime aggregate cost | Detail expire nhanh, summary cho billing. |
| `audit_events` | 2 years minimum | Compliance. **NEVER auto-delete** (chỉ explicit operator). |
| `moderation_decisions/actions` | 1 year | Pattern analysis + dispute window. |
| `knowledge_chunks` (tombstoned) | 30 days sau tombstone → hard delete | Rollback safety + hygiene. |
| Langfuse traces | 30 days | Sampling giảm volume; internal runs = source of truth. |
| `delivery_outbox` (delivered) | 7 days | Archive ra chat_events; outbox lean. |
| `processing_outbox` (done) | 7 days | Same. |

Floor: `audit_events` không cho phép < 90 days; `chat_events` không quá 7 years.

### Lifecycle Automation
Daily cron (worker container) `retention_sweeper`:
- DELETE chat_events/run_steps/model_calls quá retention (cascade FK).
- `audit_events` NEVER auto-delete.
- Tombstoned chunks: DELETE từ Qdrant + DB sau 30d.
- VACUUM ANALYZE sau bulk delete.

### GDPR / Tenant Deletion
Khi tenant `status=deleting`:
1. Stop ingest mới (adapter reject `tenant_inactive`).
2. Drain in-flight outbox → finish hoặc DLQ.
3. Hard delete: chat_events, agent_runs/steps, model_calls, moderation_*, knowledge_chunks, **Qdrant collection**, **Langfuse project**, credential handles (KMS DEK rotate/revoke).
4. Audit: `tenant_deleted` event (kept).
5. Tenant row mark deleted (giữ vì FK trong audit).
6. Hoàn thành < 30 days (GDPR).

### Backup
- pg_dump daily → B2/Storage Box, giữ 30 daily + 12 monthly.
- Qdrant snapshot weekly.
- Object storage (raw markdown): tenant control, default 1 year.

### Scale Note
100 tenants × 10K chat_events/day × 90 days = ~90M rows. Phase 1-6 single table + indexes OK. Phase 7: declarative partitioning by month.

Product-specific gates: cross-tenant SQL denial (least-privileged role), vector tenant/visibility denial, source update/delete/tombstone, disabled tool denial, audit record creation, migration downgrade, redaction tests.

- RLS → mandatory từ first tenant table (ADR-002).
- Retention → table trên (Decision 13).
- Vector deletion khi tenant delete → DELETE Qdrant collection (GDPR flow).
