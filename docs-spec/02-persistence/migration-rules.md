# Migration Rules

Alembic conventions + RLS SQL pattern cho Agent Support. SQLAlchemy 2.0 (ADR-004), PostgreSQL RLS (ADR-002).

## Alembic Conventions

- Mọi schema change qua Alembic. Không tạo/sửa table thủ công ngoài migration.
- Naming: `make migration MSG="describe change"` → revision tự generate, message kebab/descriptive.
- Review autogenerated migration **thủ công** trước khi commit — autogen miss RLS, grants, partial index, raw SQL.
- Mỗi migration phải có `upgrade()` + `downgrade()`. Nếu forward-only (vd drop column có data) → document rõ trong docstring.
- Không business logic trong migration (no data backfill phức tạp inline; nếu cần → tách script riêng + audit).
- Indexes cho: `tenant_id`, `trace_id`, source version, platform idempotency, audit queries, outbox claim.
- Test: upgrade từ empty DB + downgrade. CI gate.

```bash
make migration MSG="add tenant control plane tables"
make migrate
make migrate-downgrade   # verify reversible
make migrate
```

## RLS SQL Pattern (ADR-002)

Mỗi tenant-owned table cần: enable RLS + policy + force RLS. Đặt trong Alembic dưới dạng `op.execute(...)` raw SQL.

### Enable RLS + Policy

```sql
-- Per tenant-owned table (vd chat_events)
ALTER TABLE chat_events ENABLE ROW LEVEL SECURITY;
ALTER TABLE chat_events FORCE ROW LEVEL SECURITY;   -- áp cả cho table owner non-superuser

CREATE POLICY tenant_isolation ON chat_events
    USING (tenant_id = current_setting('app.current_tenant')::uuid)
    WITH CHECK (tenant_id = current_setting('app.current_tenant')::uuid);

GRANT SELECT, INSERT, UPDATE, DELETE ON chat_events TO app_user;
```

- `USING` → row visibility cho SELECT/UPDATE/DELETE.
- `WITH CHECK` → chặn INSERT/UPDATE sang tenant khác.
- `FORCE ROW LEVEL SECURITY` → đảm bảo policy áp dù role là table owner (trừ BYPASSRLS).

### App Role vs Operator Role

```sql
-- app_user: KHÔNG BYPASSRLS, mọi query phải có tenant context.
-- app_operator: BYPASSRLS cho incident response — audit mọi truy cập.
```

App connection dùng `app_user`. Operator queries qua `app_operator` đi qua operator API có audit (xem [../03-security/authn-authz.md](../03-security/authn-authz.md)).

### Set Tenant Context (Per-Request, BẮT BUỘC trong transaction)

```python
# SQLAlchemy 2.0 async. SET LOCAL chỉ sống trong transaction hiện tại.
async with db.begin():                         # bắt buộc transaction
    await db.execute(
        text("SET LOCAL app.current_tenant = :tid"),
        {"tid": str(tenant_id)},
    )
    # ... mọi tenant-owned query trong block này được RLS lọc
```

> ⚠️ **Leak risk:** `SET LOCAL` ngoài transaction → setting sống đến hết connection → request kế tiếp dùng lại connection từ pool → leak. LUÔN set trong `async with db.begin()`.

### pgbouncer Caveat

Nếu deploy pgbouncer trước app role: dùng **session-pool mode**, KHÔNG transaction-pool mode (transaction-pool reset session state giữa các tx, có thể phá `SET LOCAL` semantics khi nhiều statement). Đơn giản nhất: bỏ pgbouncer cho app role ở v1 single VPS (ADR-008).

### LangGraph Checkpointer (ADR-002)

AsyncPostgresSaver dùng cùng pool nhưng không tự set tenant context:
- Include `tenant_id` trong checkpoint metadata + filter app-side (cách chốt).
- Hoặc dùng service role BYPASSRLS riêng cho checkpoint tables (không khuyến nghị — mất DB-level isolation).
- `graph_checkpoint_metadata` table (RLS-enabled) map `thread_id → tenant_id` cho audit/filter.

## Migration Checklist (per tenant-owned table)

Trước khi merge:
- [ ] `tenant_id UUID NOT NULL REFERENCES tenants(id)`.
- [ ] `ENABLE ROW LEVEL SECURITY` + `FORCE ROW LEVEL SECURITY`.
- [ ] Policy `USING` + `WITH CHECK` trên `current_setting('app.current_tenant')`.
- [ ] `GRANT` cho `app_user` (không grant cho PUBLIC).
- [ ] Index trên `tenant_id` (thường composite với trace_id/timestamp).
- [ ] `upgrade()` + `downgrade()` reversible (hoặc document forward-only).
- [ ] Cross-tenant denial test thêm vào suite (least-privileged `app_user` role).
- [ ] No raw secret trong default value / comment.

## Validation Gates

```bash
make migrate              # upgrade from empty
make migrate-downgrade    # reverse
make migrate
pytest tests/isolation    # cross-tenant denial under app_user
detect-secrets scan --baseline .secrets.baseline
```

Cross-tenant denial test pattern:
```python
# Connect as app_user, SET LOCAL tenant A, query tenant B row -> expect 0 rows.
# Attempt INSERT with tenant_id = B while context = A -> expect RLS violation.
```

## Common Pitfalls

- ❌ `SET` thay vì `SET LOCAL` → leak across requests.
- ❌ `SET LOCAL` ngoài `db.begin()` block.
- ❌ Quên `FORCE ROW LEVEL SECURITY` → owner role bypass policy.
- ❌ App connection dùng owner/superuser → RLS không áp.
- ❌ Quên `WITH CHECK` → INSERT/UPDATE sang tenant khác lọt.
- ❌ Autogen migration merge thẳng không add RLS.
- ❌ pgbouncer transaction-pool mode với SET LOCAL.
