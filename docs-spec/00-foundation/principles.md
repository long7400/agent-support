# Product Principles & Design Rules

7 nguyên tắc sản phẩm + design rules dẫn xuất. Khi có xung đột giữa feature speed và principle, principle thắng.

## 7 Product Principles

### 1. Tenant boundary trước feature polish
Cross-tenant leak là rủi ro #1 của multi-tenant SaaS. Mọi feature mới phải trả lời "tenant A có thấy được data tenant B không?" trước khi merge. Isolation enforce ở DB layer (RLS, ADR-002), không chỉ app layer.

### 2. Audit/replay trước automation mạnh
Trước khi cho agent làm gì tự động, phải replay được quyết định đó từ durable records. `agent_runs` + `audit_events` là source of truth, không phải trace SaaS.

### 3. Official source trước confident answer
Agent chỉ trả lời tự tin khi có nguồn tenant-approved. Empty/stale/low-confidence retrieval → refuse/clarify/escalate. Không bịa fact crypto (giá, contract address, listing).

### 4. Policy gate trước outbound hoặc destructive action
Không message nào ra platform trước `policy_check`. Không destructive moderation (delete/ban) từ raw model text — phải qua policy matrix + idempotency + audit.

### 5. Capability manifest trước tool exposure
Model không bind trực tiếp tool list. Capability proxy filter theo tenant/role/policy. Disabled/unknown/over-budget/invalid → fail closed + audit.

### 6. Trace/redaction trước observability export
Redact secrets/PII trước khi gửi trace ra Langfuse. Redaction test là release gate. Internal audit không phụ thuộc external trace.

### 7. Small vertical slices trước platform abstraction
Telegram support + moderation shadow trước. Plugins, sub-agents, Discord, multi-source RAG sau. YAGNI cho abstraction chưa cần.

## Design Rules (dẫn xuất)

### Trust boundaries
- Tenant id không bao giờ từ untrusted request body. Resolve qua JWT/session, adapter principal, platform mapping, hoặc operator context.
- `tenant_id` immutable sau khi context được tạo.
- Adapter trust boundary ≠ admin trust boundary (PRD-012).
- Model output untrusted cho đến khi policy-checked. Tool output untrusted cho đến khi schema-validated + redacted.

### Data ownership
- Tách persistence model (SQLAlchemy ORM) khỏi API DTO (Pydantic) — không return ORM object trực tiếp (ADR-004).
- Vector backend lưu retrieval payload, không lưu product policy. Source metadata + activation luôn ở PostgreSQL.
- Secret = handle (KMS envelope, ADR-006). Raw secret không vào config/prompt/log/trace.

### Runtime safety
- Graph nodes gọi service/tool interface, không gọi raw DB/vector client.
- Mỗi capability execution re-check enablement ngay trước call (hot-disable).
- Step limits, timeouts, budgets, rate limits, backpressure — chống cost/DoS loop.
- Fail closed: thiếu thông tin/permission → deny + audit, không guess.

### Persistence discipline
- Alembic cho mọi schema change. Raw SQL trong Alembic cho RLS, roles, grants, indexes, extension.
- Review autogenerated migration thủ công.
- Mọi tenant-owned table có RLS policy + cross-tenant denial test (release gate).

### Observability discipline
- structlog, event name `lowercase_with_underscores`, variables qua kwargs (không f-string).
- `logger.exception()` cho exception (giữ traceback).
- Mọi runtime path mang: trace_id, tenant_id, run_id, platform, versions.

## Anti-Rationalization

| Cám dỗ | Thực tế |
| --- | --- |
| "App-layer filter là đủ, RLS phiền" | 1 query miss `WHERE tenant_id` = leak. RLS bắt ở DB level. |
| "Sync graph trong request cho nhanh" | Vi phạm SLO ingest <500ms + mất run khi restart. Outbox bắt buộc. |
| "SQLModel tiện hơn, giữ luôn" | RLS pattern production-grade không có cho SQLModel; mix 2 ORM = nợ kỹ thuật. |
| "Shared bot demo nhanh hơn" | Bot ban → all tenants down; rate limit chung. Per-tenant isolation. |
| "Log raw payload để debug dễ" | Leak secret/PII. Redact trước, durable audit summary internal. |
